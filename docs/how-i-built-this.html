From an empty GitHub repo to a self-updating World Cup bracket dashboard

This is the complete story of how the 2026 FIFA World Cup Bracket Dashboard was built — every command, file, and click, in order. Follow it top to bottom and you'll end up with your own interactive dashboard, hosted free on GitHub Pages, that scores itself and refreshes automatically after games finish.

It's written for someone who is not a developer. Where a step needs a tool or a concept, it's explained in plain language first. The whole thing is a single static web page generated by one small Python script — nothing to compile, no servers to run.

🧱 GitHub repo + Pages 🐍 Python generator (stdlib only) ⚙️ GitHub Actions automation 🔌 football-data.org API 🤖 Built with Microsoft Cowork

00 The big picture

Before any commands, here's the shape of the whole thing so every later step has a place to sit.

There are exactly two moving parts:

  1. A generator — one Python file (scripts/build_dashboard.py) with your picks and the live results near the top (the DATA block), and a big untouched render engine below it. Run it and it writes one self-contained web page, docs/index.html.
  2. An automation — a scheduled GitHub Action that, a few times a day, pulls finished match results from a free sports API, writes them into the DATA block, re-runs the generator, and commits the new page. GitHub Pages serves it.
input/bracket-picks.xlsx ─┐ ├─► scripts/build_dashboard.py ─► docs/index.html ─► GitHub Pages input/instructions.md ────┘ (DATA block + render) (one static file) (live website) ▲ scripts/fetch_results.py ────────────┘ (GitHub Action, on a schedule: pulls finished results into the DATA block, then rebuilds)
💡 Why one static HTML file?
Because it has zero external requests — all the CSS, JavaScript, and data are inlined. That makes hosting trivial (GitHub Pages just serves the file), it can't "break" from a missing dependency, and it opens instantly on a phone.

02 Create the GitHub repository

The repo is the home for the code and the live website. Public, so GitHub Pages is free.

Option A — the website (easiest)

  1. Go to github.com/new.
  2. Repository name: e.g. my-wc2026-bracket.
  3. Set it to Public.
  4. Do not add a README or .gitignore yet — you'll push your own.
  5. Click Create repository.

Option B — the GitHub CLI

gh repo create my-wc2026-bracket --public --clone
💡 What "public" means here
Your picks and results are just for fun, and a public repo is what makes free GitHub Pages hosting work. No secrets ever go in the code — the one API token you'll add later is stored separately and encrypted (Step 12).

04 The generator — DATA block + render engine

This is the heart of the project. Copy the Python generator out of the build kit into scripts/build_dashboard.py. You'll only ever edit the top of it.

The file has two clearly separated parts:

  • USER DATA (the top ~60 lines) — your name, tiebreaker, the 16 first-round fixtures and your picks, and the finished results. This is the only part you change.
  • RENDER ENGINE (everything below the banner) — the layout, colours, bracket geometry, scoring maths, hover cards, reading modes, the side-navigation rail, etc. Never edited — it's what guarantees every entrant gets the identical design.

A trimmed look at the DATA block so you can see its shape:

ENTRANT="Eric Lam"          # the name on the dashboard
TIEBREAKER=4               # total goals in the Final (your guess)
REFRESHED="July 2, 2026 · 11:40 AM PT"  # "last updated" stamp

# The 16 Round-of-32 games: (match code, date, teamA, teamB, YOUR pick)
R32=[("M74","Mon 6/29","Germany","Paraguay","Germany"),
     ("M75","Mon 6/29","Netherlands","Morocco","Morocco"),
     # ... 14 more ... ]

# Finished results: match code → (goalsA, goalsB, winner, note)
RES={"M74":(1,1,"Paraguay","4–3 pens"),
     "M75":(1,1,"Morocco","3–2 pens")}

# Your later-round picks
CHAMP="England"; RUNNER="France"

# ════ RENDER ENGINE — do not edit anything below this banner ════
💡 The scoring rule, in one sentence
Each correct advancing pick earns points that double every round (R32 = 1, R16 = 2, QF = 4, SF = 8, Champion = 16; 80 max). A game not yet in RES is treated as pending and counts toward "still live" points, never as a loss.

05 Put in your picks

Open scripts/build_dashboard.py and edit only the DATA block to match your Excel's "My Bracket" tab.

  • Set ENTRANT and TIEBREAKER.
  • Fill R32 with your 16 fixtures and picks.
  • Set your later-round picks (R16_PICK, QF_WIN, SF_WIN, CHAMP, RUNNER).
  • Leave RES empty (or with only genuinely-final games) — the automation fills it as matches play.
🤖 The shortcut most people will use
Instead of editing Python by hand, hand the whole kit to Microsoft Cowork and say "follow this", then upload your Excel. Cowork reads your "My Bracket" tab, looks up the results, fills the DATA block for you, and runs the generator untouched.

Expect external refs: 0.

07 Turn on GitHub Pages

This is the free hosting. One setting, and your page is live on the internet.

  1. In your repo on github.com, open Settings → Pages.
  2. Under Build and deployment → Source, choose Deploy from a branch.
  3. Pick branch main and folder /docs. Click Save.

Within about a minute it's live at:

https://<you>.github.io/my-wc2026-bracket/

Every time a push changes docs/, Pages re-deploys automatically. That single fact is what makes the automation in the next steps "just work."

08 The sync script — fetch_results.py

This is the brain of the automation. It reads finished games and writes them into the DATA block, then rebuilds the page. It runs the same whether triggered by the schedule or by you.

What one run does, in order:

  1. Reads your R32 fixtures straight out of build_dashboard.py, so it always matches your bracket.
  2. Gets finished matches from the web (the football-data.org API), or from a local --input results.json file for testing.
  3. Normalises team spellings using team_map.json (Step 9).
  4. Matches each finished game to one of your bracket games by the pair of teams.
  5. Writes changed/new results into RES / UPCOMING / REFRESHED, then re-runs the generator.

It has handy flags for safe experimentation:

python scripts\fetch_results.py --dry-run             # show what WOULD change, write nothing
python scripts\fetch_results.py --input results.json  # use a local file instead of the web
💡 Built-in safety
It only ever writes finished games, it's idempotent (if nothing new is final, it changes nothing), and with no token and no --input it exits cleanly as a no-op — so the automation never shows a red ❌ just because setup isn't finished.

09 The team-name map — team_map.json

Different sources spell countries differently. This tiny file translates the feed's spelling into your bracket's spelling so games match up.

{
  "Bosnia and Herzegovina": "Bosnia & Herz.",
  "Côte d'Ivoire":          "Ivory Coast",
  "Korea Republic":         "South Korea"
  // … left side = feed spelling, right side = your bracket spelling …
}

If a result ever fails to match, the fix is almost always: add one line here.

10 The GitHub Action — sync-results.yml

A GitHub Action is a small recipe GitHub runs for you on their servers. This one checks out the repo, runs the sync script, and commits the refreshed page if anything changed.

Put this in .github/workflows/sync-results.yml:

name: Sync World Cup results

on:
  schedule:
    # Times are UTC. Three runs, timed just after each block of games ends.
    - cron: "0 22 * * *"    # ~6:00 PM ET
    - cron: "30 1 * * *"    # ~9:30 PM ET
    - cron: "0 6 * * *"     # ~2:00 AM ET
  workflow_dispatch: {}      # adds a manual "Run workflow" button

permissions:
  contents: write            # lets the job commit the refreshed page back

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: {python-version: "3.12"}
      - name: Fetch results and rebuild
        env:
          FOOTBALL_DATA_TOKEN: ${{ secrets.FOOTBALL_DATA_TOKEN }}
        run: python scripts/fetch_results.py
      - name: Commit if the dashboard changed
        run: |
          if [[ -n "$(git status --porcelain docs/index.html scripts/build_dashboard.py)" ]]; then
            git config user.name "github-actions[bot]"
            git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
            git add docs/index.html scripts/build_dashboard.py
            git commit -m "chore: auto-sync World Cup results [skip ci]"
            git push
          fi
💡 The mental model
There's no server you rent or babysit. GitHub itself wakes up on the schedule, runs these steps on a throwaway Ubuntu machine, and shuts it down. You only pay attention if you want to (the "Run workflow" button).

11 Get a free API token

The script needs a key to read match results from football-data.org. The free tier is plenty for a tournament.

  1. Go to football-data.org/client/register and sign up (free — no card).
  2. Your API token arrives by email and shows in your account.
  3. The dashboard uses competition code WC (FIFA World Cup) and only reads finished matches.
⚠️ Rate limits
The free tier limits how often you can call the API — which is exactly why the schedule runs just 3×/day instead of hourly. Don't crank the schedule up.

12 Add the token as a repository secret

Secrets are stored encrypted by GitHub and handed to the workflow only at run time — they're never written into your code.

  1. Repo Settings → Secrets and variables → Actions.
  2. Click New repository secret.
  3. Name: FOOTBALL_DATA_TOKEN (exactly this — the workflow reads that name).
  4. Secret: paste your token → Add secret.

That's the link between "the workflow" and "the API." Without it the workflow still runs green, it just does nothing.

13 Understand (and tune) the schedule

The three cron lines are timed to fire shortly after each daily block of matches ends, so results land within ~30–60 minutes of the final whistle — never mid-game.

RunApprox. time (ET)Catches kickoffs…Cron (UTC)
☀️ Afternoon~6:00 PMnoon–3 PM0 22 * * *
🌆 Evening~9:30 PM5–6:30 PM30 1 * * *
🌙 Late night~2:00 AM8–11 PM0 6 * * *

A cron line is five fields: minute hour day month weekday. 0 22 * * * means "at minute 0 of hour 22 (UTC), every day." Summer Eastern Time is UTC−4, so 22:00 UTC = 6:00 PM ET. Change these numbers to shift when it runs.

14 Test the automation

You don't have to wait for the schedule — trigger it by hand.

  1. Repo Actions → Sync World Cup results → Run workflow → Run.
  2. Watch the run. A green ✅ with a log line like Source: football-data.org … means it's wired up.
  3. If no games are newly final, it correctly reports No new finished games to apply — that's success, not failure.
✅ Done — it now runs itself
From here the workflow fires on its schedule, pulls finished games, rebuilds docs/index.html, and pushes — which re-deploys Pages automatically. The only "manual" parts were this one-time setup and the optional Run-workflow button.

15 A real bug — and the fix worth reading

This actually happened during the build, and it's a great lesson in trusting real data over assumptions.

The first live automation run changed two penalty-shootout games from "1–1 (4–3 pens)" and "1–1 (3–2 pens)" into nonsensical-looking 4–5 and 3–4 scorelines. The winners were still right, but the scores looked wrong.

🐞 The root cause
The football-data.org API folds the penalty-shootout kicks into the full-time score. A game that was 1–1 and won 4–3 on penalties is reported as 5–4 in the fullTime field. The original code didn't expect that, so it wrote the folded score and dropped the "pens" note.

The fix: detect a shootout via the separate penalties field, subtract the shootout tally to recover the true level score (5−4 pens → 1–1), take the winner from the API's own winner field, and rebuild the clean "4–3 pens" note. A guard was also added so a note-less value can never silently overwrite a curated shootout result again.

💡 The takeaway
Real feeds have quirks the docs don't mention. When automated data looks wrong, inspect the actual payload and reconcile the numbers — don't assume the source matches your mental model.

16 The v7 UI polish

Two finishing touches were added to the render engine (and to the reusable kit, bumping it to v7).

  • A top "last refreshed" indicator — a small live chip in the header mirroring the footer timestamp, so the last sync time is visible without scrolling.
  • A sticky left side-navigation rail with a scrollspy that highlights your current section, smooth jumps, and a mobile tap-drawer.
💡 The one tricky CSS detail
The rail sits in a two-column layout, but the bracket map must never get its own horizontal scrollbar. The trick: .content{min-width:0} lets the content column shrink so the bracket keeps its own overflow behaviour, and body{overflow-x:hidden} is the final guard. Below 1200px the layout collapses to a single column and the rail becomes a 📑 Contents drawer.

Because these live in the render engine, they need no per-entrant data — every rebuild reproduces them automatically. That's the whole point of the DATA-vs-engine split.
