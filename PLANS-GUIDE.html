1 Safety net & drift alarm Do this first

PLAN-ci-drift-guard-and-tests.md
Stupidly simple: Three times a day, a robot (GitHub Actions) edits your Python code with find-and-replace and pushes the result — with nobody checking its work. Once already (issue #2), a fix got silently erased this way. This plan adds tests plus a tripwire: if the published page ever stops matching what the generator produces, GitHub turns red and tells you. It also fixes a real bug I found where a group-stage result could overwrite the Final's result if the same two teams meet twice.

A tests/ folder, a sample data file, and one new workflow file .github/workflows/ci.yml. One small bug fix in fetch_results.py.

  4. Commit & push. Go to the repo's Actions tab — a new "CI" workflow should run and pass.
  5. Optional proof it works: hand-edit one letter in docs/index.html, push, and watch CI fail. Revert after.
Read PLAN-ci-drift-guard-and-tests.md and execute it exactly. Don't skip the pair-collision fix or the edge cases section. Verify every acceptance criterion before finishing.

3 Move the data out of the code

PLAN-json-data-layer.md
Stupidly simple: Right now your picks and the live results are typed inside the Python script itself, and the sync robot updates them by doing regex find-and-replace on its own source code. That's like a surgeon operating on themselves — it works until it really doesn't. This plan moves all the data into plain data/*.json files. The script reads them; the robot writes them. Nobody edits code anymore. This is also the key that unlocks Plan 6 (other people's brackets) — you can't have "everyone's picks" while the picks live inside one Python file.
What gets created / touched
New: data/picks.json, data/live.json, data/topology.json. Edited: both scripts, the sync workflow, README.
How to execute
  1. Only after Plans 1 & 2 are merged — the tests are what make this refactor safe.
  2. Paste the prompt below.
  3. The magic checkpoint: after the change, python3 scripts/build_dashboard.py must produce a byte-identical docs/index.html (git diff docs/index.html shows nothing). If it doesn't, the data was transcribed wrong — the plan says to fix the JSON, never the engine.
  4. Trigger a manual sync from the Actions tab and confirm the commit now touches data/live.json + docs/index.html — and no longer touches the Python script.
Read PLAN-json-data-layer.md and execute it exactly. The non-negotiable gate: after moving the data to JSON, the regenerated docs/index.html must be byte-identical to the committed one before you proceed. Verify all acceptance criteria.

You're done when: grep -c "re.sub" scripts/fetch_results.py prints 0, and a sync run commits JSON instead of editing Python.
